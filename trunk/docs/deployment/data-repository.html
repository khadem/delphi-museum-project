Deploying Delphi: Data Repository

This is a draft page, in ongoing development. It is still rough, but we are exposing it to the community for review and to help anyone considering a Delphi deployment. We are actively updating all of the documentation, so it is best to use this resource online, rather than printing out a version.

This is an overview of the data repository in Delphi, an open source software system that helps open up access via the Web to collections of artifacts held by museums, historical societies, archeological repositories and similar heritage institutions.  Delphi uses artificial intelligence and natural language processing to make it more effective and enjoyable to find individual artifacts in collections, and to uncover relationships between artifacts, than is possible through traditional searching and browsing.

Types of data stored in the repository

The data repository stores two types of data:
  • Data about artifacts in the collections ("metadata").

    Purposes of this data:  This data is displayed when the Delphi collections browser provides details about individual artifacts.

    In addition, this data is "mined," via natural language processing, to help build a set of concepts that more fully describe the collection and then to associate those concepts with each of the individual artifacts in the collections.

    Source of this data:  In the instance of Delphi used by PAHMA, data about the artifacts in its collections ("metadata") comes directly from that museum.

    Future instances of Delphi deployed elsewhere will similarly obtain this metadata from their own museums or heritage institutions.  In each of those instances, the format of this metadata will vary.  This format depends both on how any specific institution organizes the metadata about its collections, and potentially also on the collections management system it uses to store that data.

    As a result, the code and/or configuration of Delphi's back-end tools, which load this metadata into the data repository, will need to be customized for each instance.  However, the database schemas used to store this metadata in Delphi are relatively generic, and might not need to be changed, or require only minor changes.

  • Data about the concepts associated with these artifacts (a "semantic index").

    Purpose of this data:  This data helps users of Delphi's collections browser locate artifacts by concepts, and help uncover relationships between artifacts.

    Source of this data: This data is generated by Delphi's back-end tools (below), based on the data provided by the museum.

    Future instances of Delphi deployed elsewhere will need to build their own mental models ("ontologies") of their data facets and concepts.

Components

The data repository currently consists of the following components:

1. MySQL

The data repository for the instance of Delphi currently used by PAHMA is MySQL, an open source relational database management system (RDBMS).  MySQL is available in two flavors - MySQL Community Server (free), and MySQL Enterprise (commercial, packaged with an enhanced set of support services).

Delphi can also potentially use other relational database software in place of MySQL.  This is because Delphi's PHP-based code makes generic database calls via a PHP library, Pear::MDB2, that supports a number of different types of databases.

The Delphi team had considered other types of data repositories, such as a "triple store" that is optimized for storing and retrieving semantic data represented via the Resource Description Framework (RDF), but chose a relational database primarily on the basis of performance.

Deployment details for the PAHMA instance of MySQL

This is a quick and dirty initial pass at this section, with refinement needed.

MySQL is provided by the IST database team, as an extra-cost option to the IST Unix Web Farm hosting environment at UC Berkeley.

Delphi uses the MySQL Basic service, which "is appropriate for projects which have no restricted data, modest storage requirements, and no complex database design issues. It includes a single production environment with limited support."  For more information regarding this service,  see http://ist.berkeley.edu/services/catalog/database/mysql

In the three-tier model of the IST Unix Web Farm, it is possible to have separate instances of databases, such as MySQL databases, available to each of the tiers.  This way, development and quality assurance work that changes database contents can be carried out without affecting the production database.

In the current instance of Delphi on the Web Farm, there are two instances of MySQL: one shared by the development and quality assurance environments, and another specific to the production environment.

The phpMyAdmin console is provided by default in the IST Unix Web Farm hosting environment at UC Berkeley *VERIFY THIS*, and is often provided in other web hosting environments that support PHP.  phpMyAdmin can also be obtained independently from http://www.phpmyadmin.net
